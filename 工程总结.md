## 1. 项目定位

本项目是 SpecBench 的任务构建工作台，从真实 Python 开源仓库中构建多文件、全项目重建型 SWE-E2E benchmark task。候选 agent 仅能看到公开行为规格和通用实现指令，不能看到原始源码、原始测试、oracle、分数、过往轨迹或其他候选解。

项目的核心不是追求测试数量本身，而是确保每一个计分点同时满足三条原则：

1. **Like a developer**：spec 必须像库作者写给开发者的文档，不能暴露 benchmark 的审计语言和内部元数据。
2. **Spec-driven**：每条保留测试都必须映射到 spec 的真实章节，候选人仅阅读 spec 就能推导出预期行为。
3. **Behavioral**：测试检查公开可观察行为，不检查私有字段、内部 map、`repr`、精确异常文案或实现特有 shape。

## 2. 总体架构

流水线由一个主调度 agent 和四个阶段 specialist skill 组成：

```text
candidate-selector -> spec-writer -> test-filter -> cleanroom evaluation -> task-judge
       |                  ^              ^                                  |
       +-> RETIRED        +-- spec gap --+-- filter/fairness correction ----+
```

| 阶段 | 责任 | 核心产物 | 主要出口条件 |
|---|---|---|---|
| Stage 1 | 候选库筛选 | `filter_notes.md`、`CANDIDATES.md` 记录 | 规模、多公开投影、docs-test 对齐和可隔离性通过 |
| Stage 2 | 公开行为规格撰写 | `spec_vN.md`、公开 `spec.md` | 公开 API surface 已穷举，11 项 spec 检查通过 |
| Stage 3 | 测试改写、过滤、生成和分层 | `spec_test_map.md`、`kept_nodeids.txt`、`taxonomy.jsonl` | oracle 可追溯、行为性、dummy gate、reference gate 和覆盖配额通过 |
| Stage 4 | cleanroom 候选实现与评分 | 候选代码、轨迹、`score_result.json` | 候选只看公开 packet，评分环境隔离，导入来源可证明 |
| Stage 5 | anti-cheat、可解性、公平性与失败诊断 | `judge/diagnosis_report.md`、weakness 记录 | Gate A-D 通过，或明确回退到 Stage 2/3/4 |

每个在制任务由 `PIPELINE_STATE.md` 维护状态、当前 todo、`spec_iter`、`filter_iter`、`eval_iter` 和 History。状态不匹配时不应派发下一阶段；子 agent 返回后没有更新状态和 History，视为未正常完成。

## 3. 各阶段工程细节

### 3.1 候选库筛选

当前硬门槛包括：纯 Python 源码通常不少于 3,000 LOC；不能用单个小文件完成；必须有一个共享事实源和至少两个独立公开投影；测试不能主要依赖网络、平台或精确快照；不选高饱和标准模式；公开文档与可用测试的投影类型必须对齐。

Stage 1 还要执行私有 import 预筛。超过 30% 测试文件在模块顶层导入私有符号时，标记 `HIGH_RISK`，不会立即淘汰，但意味着 Stage 3 需要大量公开 API 改写或提前进入生成测试路径。

### 3.2 行为规格

Spec writer 可以阅读源码和原始测试，候选 agent 不可以。写作前要从 `__init__.py`、`__all__`、公开 docstring、README、官方文档、例子和 CLI help 给出完整公开 surface 清单，再对每个符号分别判断：

- 是否属于对用户承诺的公开行为。
- 是否是高级工程师不能从通用常识自行推导的库特有信息。

Spec 必须包含 Product State Model、至少 3 个公开状态投影、至少 6 条跨视图不变量、明确失败路径、至少 1 个端到端工作流、Non-Goals 和不暴露 fixture shape 的 Evaluation Notes。内部审计 header 在发给候选 agent 前必须剥离。

### 3.3 测试 oracle 构建

Stage 3 对每个 nodeid 分别审计，不按文件粗放处理。一条测试只有在“换一种内部实现仍能通过”且“只读 spec 就能推导预期”时才能保留。

工程上有两条路径：

- **Track A：上游测试清理。** 修复共享 fixture/import，将能保持原行为意图的测试改写为仅使用公开 API，然后完成逐函数去留映射。
- **Track B：覆盖引导生成。** 当 Track A 丢弃超过 50% 文件、全部丢弃、保留少于 30 个 nodeid，或缺少 integration/system 覆盖时启动。生成 agent 可执行 reference 观察输入输出，不可阅读 reference 源码。

当前 `test-filter` 的最低配额为：Cross-View Invariants 至少 5 条，Error Semantics 至少 3 条，主工作流至少 3 条，其他每个 H2/H3 至少 3 条；合并 oracle 原则上至少 50 个可计分测试。少于 50 只能在每节配额都满足、继续生成会引入循环断言时例外放行，并必须记录 caveat。

所有 oracle 还要通过 dummy gate、reference gate，并分类为 `atomic`、`integration`、`system_e2e`。`spec_test_map.md` 是筛选过程的主证据，不是事后补写的汇总文件。

### 3.4 Cleanroom 和评分

`harness/run.py` 会创建独立 workspace，只复制 `spec.md`/`prd.md`、显式公开 packet 和可选 starter，创建空 `solution/`，并拒绝 manifest、reference、score report 等禁止路径。该脚本实现了公开 packet 与隐藏评测资产的物理分离。

`harness/score_pytest_original.py` 复制 oracle 测试树，通过 `--remove-path` 移除原包，再用 `PYTHONPATH` 指向 reference 或 candidate solution。它按测试文件分组运行 pytest JSON report，将参数化 nodeid 归一化后映射到 taxonomy，输出总分、分层分数和跳过后通过率。

每次有效候选评分必须先执行导入来源预检，证明 `<pkg>.__file__` 指向 candidate 输出而不是源仓库或系统已安装包。没有预检字面输出的 diagnosis report 按当前 judge skill 是结构上无效的。

### 3.5 Judge 门禁

Stage 5 先审查评测工具，再解释分数：

- Anti-cheat：检查导入来源和完整轨迹，发现读取源码、测试、评分或前序尝试时作废该 run。
- Solvability：reference 在专用依赖环境中必须通过 oracle，收集错误和缺依赖不能归因于模型。
- Gate A：抽查 `spec_test_map` 映射是否真实可推导。
- Gate B：抽查失败簇是公开行为缺口，还是 verifier 过度约束。
- Gate C：对 generated-only oracle 至少人工/独立抽查 5 条。
- Gate D：核对所有 spec H2/H3 的覆盖，给出 `FULL`、可接受 `PARTIAL` 或必须修复的 `GAP`。

候选失败要区分根因与级联失败。导入缺失导致的大量 integration/system 失败不能伪装成 composition signal；只有无法由底层缺失解释的跨组件状态偏差，才是真正的组合能力证据。

## 4. Agent 编排与审查责任

当前可执行的编排口径如下：

1. 主 agent 直接按 `PIPELINE_STATE.md` 派发各阶段 specialist agent，派单中明确要求子 agent 先读当阶段 `SKILL.md`。
2. 每个阶段需要独立的执行责任边界，但不要求每阶段再叠加一个额外“审查 agent”。默认是 specialist 交付产物，主 agent 按三原则和交接门禁审查。
3. Stage 2 可使用独立 checker agent。checker 必须读 `spec-writer/SKILL.md`，只输出可操作的问题清单，直接返给 writer 修订；不让人在每次 writer-checker 往返中重新完整审查。
4. 机器流程的目标是让 task 先通过 judge，再由人对整体任务是否符合 Like a developer、Spec-driven 和 Behavioral 做最终把关。人工最终审查不能被 judge 替代，但也不应成为每个机械迭代的同步阻塞点。

## 5. 已确认的流程决策

### 5.1 Stage 5 因 oracle 不公平回退到 Stage 3

当 PRD/spec/公开 packet 没有任何变化，Stage 5 只发现 retained oracle 不公平时，不重新生成 cleanroom candidate solution。原候选实现作为不可变快照复用，Stage 3 修复 oracle 后，用同一份 candidate 重跑评分和 Stage 5 judge。

这个决策的前提是候选 agent 当时看到的所有公开输入完全不变。只要 spec、依赖约束、starter、运行指令或公开协议发生变化，就必须开新 cleanroom run，不能让旧 candidate 享受事后信息。

### 5.2 上游原始测试全部被筛掉

上游测试全部被筛掉不会自动导致 task 淘汰。只剩 LLM 生成测试时，任务进入 `generated_only` oracle 路径，必须满足当前 `test-filter` 的绝对数量和分节配额，通过 dummy/reference gate，并在 Stage 5 额外抽检至少 5 条生成测试。

当前仓库中 `diskcache-fullrepro-001` 是这条路径的有效实例：63 条 generated-only 测试，dummy 0/63，reference 63/63，Gate C 抽检和 Gate D FULL 均已记录。

### 5.3 Unit 和 system 都是 100%

候选在 unit/atomic 和 system_e2e 上都达到 100% 时，任务先保留，不因满分自动淘汰。Judge 必须执行饱和度检查，当候选实现文件数显著少于原包且仍为 100% 时，标记 `trivially-solved` 和 `saturated-candidate-score`，将 benchmark 区分度不足写入 manifest caveat。是否在后续数据集筛选中移除，等待跨任务、跨模型结果再做决定。

`hypercorn-fullrepro-001` 是当前实例：reference 52/52，candidate 52/52，已标记饱和、过于容易、system_e2e 仅 2 条和服务协议覆盖有限等 caveat，状态应是 `QUALIFIED_WITH_CAVEATS` 而不是无条件高质量任务。

### 5.4 Spec 检查的迭代计数

团队已确认的运行口径是：Stage 2 完成后，checker agent 将问题清单返回 writer 的阶段内检查不记入防止无限循环的正式迭代次数。迭代计数用于限制任务在不合适的方向上反复跨阶段回退，而不是限制 writer 在首次交付前修正检查问题。

应记数的是已经完成阶段交接后，由 judge 或下游门禁触发的正式回退，例如 spec gap/error 返回 Stage 2、oracle 公平性问题返回 Stage 3、cheat/环境问题返回 Stage 4。

## 8. 当前未闭环问题

### 8.1 本地整合状态尚未发布

本地 Git 基线只跟踪 cookiecutter 和 vcrpy 两个 task；当前工作区共有 14 个已跟踪文件被修改、41 个未跟踪条目，其中包含其余 28 个 task 目录。因此“本地已整合 30 个任务”不等于“GitHub main 已发布 30 个任务”。

### 8.2 状态源不一致

30 个子任务 `MANIFEST.json` 都自报 `QUALIFIED`，但根级 `CANDIDATES.md` 将 4 个标为 `QUALIFIED_WITH_CAVEATS`、2 个标为 `HOLD_EVIDENCE`。根级 `MANIFEST.json` 只包含套件策略，没有机器可读的 task registry。发布前必须选定单一状态源，并用校验器阻止根级台账、子 manifest 和 judge verdict 漂移。

### 8.3 Provenance 证据不完整

19 个任务的 `scorer_isolation.recorded_from_artifacts` 为 true，7 个显式为 false，4 个没有该字段。显式为 false 的是 cerberus、cookiecutter、lark、marshmallow、pyparsing、tomlkit 和 vcrpy。这不代表这些任务一定无效，但表示不能仅凭后补文档把历史分数当作完整 cleanroom 证据。

cookiecutter 的标准化 diagnosis 写明 reference 为 `0/0`，preflight 仅是报告中的不可本地验证路径；因此继续保持 `HOLD_EVIDENCE` 是必要的。vcrpy 的 preflight 明确写着 `not recorded`，同时有 Gate D GAP 和低于 50 的问题，不应按严格当前标准声称 QUALIFIED。

### 8.4 覆盖口径仍有 GAP

子 manifest 中 18 个明确记录 Gate D FULL，4 个记录 GAP，1 个记录 PARTIAL，7 个缺少统一 verdict 字段。当前明确 GAP 包括 cerberus 的 Required Fields、marshmallow 的 Utility Helpers/其他低覆盖节、pyparsing 的 Diagrams、vcrpy 的多个行为节和全局数量底线。只有在无法继续生成 spec-derivable 且非循环的测试时，才能将 GAP 作为显式 caveat 接受。

### 8.5 流程文档存在相互冲突

当前不能回避的冲突包括：

- `task-synthesizer` 写 `spec_iter > 2` 退出，`spec-writer`、`task-judge`、状态机模板和中文说明的主体段落写 `> 3`。
- `test-filter` 要求 Error Semantics 至少 3 条，`task-synthesizer` 交接检查却写 Cross-View Invariants/Error Semantics 都至少 5 条。按“阶段 specialist skill 是阶段权威”的现有约定，Stage 3 应先用 3，但文档必须统一。
- Stage 3/S4 状态前置要求 reference 100%，`task-judge` 的 Solvability 硬门槛却是至少 95%。当前已整合任务台账都声明 reference 100%，发布门禁宜继续使用更严的 100%。
- 中文工作流的 Stage 4 Analysis 建议饱和时回到 Stage 2，judge skill 和本次团队决策则是先保留并标签。当前实际口径以“保留 + 标签 + caveat”为准。
- README 要求人在每个阶段都检查，团队聊天确认的目标是先让 writer/checker/judge 自动闭环，通过 judge 后再人工做整体原则审查。这一责任分配尚未写回权威 skill。
- `spec-writer` 当前将任何检查失败都记为 `spec_iter += 1`，与团队新确认的“Stage 2 内部 checker-writer 修订不记入正式循环”口径冲突。

这些不是文字细节，而会直接导致同一 task 被不同 agent 作出不同退休、补测试或重跑决策，应在下一次 skill 更新中一次性统一。

### 8.6 产物可移植性不足

28 个 task manifest 中存在指向 `/root/...` 的绝对路径，共检出 137 个绝对路径值，多数指向 `Bmk-Lizhiqian`、`new-e2e` 和候选 run 目录。这些路径可作为当前机器上的 provenance 线索，但不能作为可移植交付合同。正式发布时应改为仓库相对路径、内容哈希和可重建 run ID，并用一次冷启动复现验证不依赖当前工作站。

### 8.7 旧 harness 与当前方向并存

`harness/score.py` 仍包含 mini Redis/Bitcask 等历史 rubric 评分路径，根级 manifest 已将旧 mini-task/unit-system-gap 方向标记为 retired。新任务应使用 `score_pytest_original.py` 及各 task 明确记录的 scorer，不应让旧评分器成为新流水线的默认入口。

## 9. 发布前收口顺序

1. 先统一 skill 和状态机中的迭代计数、Error Semantics 配额、reference pass rate、饱和处理和人工审查时点。
