# Benchmark 任务设计核心路线

这类 benchmark 的本质不是“从头实现一个开源项目”，而是把真实 GitHub 项目抽象成一个可评测的任务包。

核心交付物主要是：

```text
prd.md        给模型看的公开任务需求
rubric.json   给评测器看的测试标准
```

最重要的关系是：

```text
PRD 是自然的公开产品需求。
rubric 是基于 PRD 的验收测试。
rubric 不能只是 PRD 的逐条翻译。
```

如果 `rubric.json` 只是把 `prd.md` 每一句翻译成一个测试，模型只需要逐条照做，就可能通过全部测试。这种任务区分度不够，也不能很好观察 E2E gap。

## 1. PRD 和 rubric 的正确关系

需要同时满足两个原则：

1. rubric 的所有测试都能在 PRD 中找到原型。
   - 不能出现 PRD 没提过、测试突然要求的隐藏功能。
   - 测试可以组合多个 PRD 行为，但不能凭空发明需求。

2. PRD 不能写得像测试答案。
   - 不能把所有命令、返回值、边界条件、测试顺序都列得过细。
   - PRD 应该像真实项目抽象出的产品说明，而不是 rubric 的自然语言版。

更合理的关系是：

```text
PRD 写能力和行为边界。
rubric 写验收场景和组合压力。
```

例如：

```text
PRD:
Failed LOAD must not corrupt the previously loaded graph.

rubric:
先 LOAD 一个有效图，再 LOAD 一个 trailing comma 坏文件，
随后 INFO / LIST / UNRESOLVED 仍必须反映旧图。
```

这里 rubric 来自 PRD，但不是简单翻译，而是构造了一个跨步骤、跨状态的系统场景。

## 2. 一个任务是怎么做出来的

可以按 8 步理解：

1. 选真实 GitHub 项目
   - 找一个真实开源项目，例如 `minishell`、`bgraph`。
   - 判断它是否有真实业务逻辑、状态变化、错误处理和组合行为。
   - 不选只有简单输入输出、没有系统状态的项目。

2. 抽象成小型任务
   - 不复刻整个项目。
   - 只抽出“小而完整”的核心能力。
   - 任务最好能用 CLI 或库接口表达，方便自动评测。

3. 写自然 PRD
   - 描述公开行为、功能边界和系统不变量。
   - 不直接暴露测试路径。
   - 不把每个边界 case 都列成答案。

4. 写 requirement map
   - 把 PRD 中的需求拆成编号，例如：
     - `REQ-load-success`
     - `REQ-error-atomicity`
     - `REQ-pipe-dataflow`
   - 说明每个需求会被哪些 unit/system 测试覆盖。

5. 写初版 rubric
   - unit 测单点功能。
   - system 测组合行为、跨命令状态、错误恢复、顺序敏感。
   - 每个测试必须能回溯到 PRD 原型。

6. 写 reference 并跑通
   - reference 是标准答案，用来证明 PRD 和 rubric 可实现。
   - reference 必须达到 unit 100%、system 100%。

7. 跑多个模型 candidate
   - 用 code agent 或 API 模型生成实现。
   - 跑 rubric 得到 pass rate 和失败 case。
   - 根据真实失败现象判断任务是否有区分度。

8. 动态调整 rubric
   - 如果所有模型都 100/100，说明 system cases 不够强。
   - 如果 reference 过不了，说明 PRD/rubric 有歧义或测试不公平。
   - 如果模型 fail 的点确实来自 PRD 中的系统不变量，就可以增强相关 system cases。

## 3. PRD 应该怎么手写

PRD 不应该像测试文件，而应该像产品需求文档。

推荐结构：

```md
# MiniXXX Public Product Packet

## Overview
说明任务是什么，来自哪个真实项目，最终要实现什么程序。

## Feature Set
列出 5-8 个核心功能模块。

## Global Invariants
说明任何情况下都必须成立的系统规则。

## Command Input Model
说明程序怎么运行、怎么读取输入。

## Data / File Format
如果有文件输入，说明文件格式。

## Commands / Public API
说明支持哪些命令或接口，以及它们的大体行为。

## Error Behavior and Recovery
说明错误发生后状态是否保持，错误输出到哪里。

## Non-Goals
明确哪些功能不用做，避免任务过大。

## Evaluation Style
说明会用 unit/system cases 检查公开行为，但不暴露测试细节。
```

手写 PRD 时注意：

```text
要写真实项目抽象出的能力，不写测试脚本。
要写系统不变量，不写所有 case 的答案。
要描述行为边界，不绑定内部实现。
```

PRD 中可以写：

```text
Failed operations must not corrupt previously valid state.
Outputs should be deterministic and ordered where applicable.
Pipeline output should feed later pipeline stages.
```

PRD 中不建议写：

```text
先输入 A，再输入 B，必须输出 C、D、E。
遇到 trailing comma 这个 exact case 必须输出某一行。
第 13 个测试会检查某个具体组合。
```

## 4. rubric.json 应该怎么手写

`rubric.json` 是测试清单。每个 case 建议包含这些字段：

```json
{
  "id": "MBGS013",
  "layer": "system",
  "system_dimension": "error_atomicity",
  "category": "trailing_comma_load_preserves_graph",
  "requirement_refs": ["REQ-load-error-atomicity", "REQ-info-basic"],
  "description": "A malformed LOAD fails and preserves the previous graph.",
  "weight": 8,
  "test_code": "...",
  "expected_output": "OK\nTrue\nname=app"
}
```

字段含义：

```text
id                 测试编号
layer              unit 或 system
system_dimension   system case 的系统维度
category           测试类别
requirement_refs   对应 PRD / requirement_map 中的需求编号
description        人能看懂的测试说明
weight             权重
test_code          实际测试代码
expected_output    期望输出
```

unit 和 system 的区别：

```text
unit:
检查一个局部功能是否成立。
例如 LIST 是否排序、echo 是否输出、export 是否设置变量。

system:
检查多个功能组合后系统状态是否仍然正确。
例如 failed LOAD 后旧图是否保留，export 后变量是否进入 pipeline，
REMOVE 后 unresolved 和 RDEPS 是否一致。
```

常用 system 维度：

```text
state_accumulation          状态是否跨命令保留
cross_feature_dataflow      一个功能的输出是否正确进入另一个功能
global_invariant            全局规则是否一直成立
error_atomicity             出错是否不污染状态
operation_order_sensitivity 操作顺序是否影响结果
boundary_crossing           边界组合是否正确，例如管道 + 重定向
durability_reload           跨进程或跨重载后状态是否一致
```

## 5. 动态跑模型调整 case

可以先部署 code agent 进行初步测试，根据 pass rate 动态调整任务。

可用的 agent 类型：

```text
工业级 code agent:
Claude Code、Codex、OpenCode
可以通过 subagent 方式并行生成 candidate。

科研型 code agent:
SWE-Agent、Mini-SWE-Agent、OpenHands
适合做更完整的 agentic coding 流程评测。
```

推荐循环：

1. 用 PRD 生成 candidate。
2. 跑 rubric。
3. 观察 unit/system pass rate。
4. 分析失败是否来自真实 PRD 要求。
5. 如果失败合理，增强同类 system cases。
6. 重新跑 reference，确保 reference 仍 100%。
7. 再跑 candidate，确认 gap 是否稳定。

注意：

```text
不能为了制造 gap 加隐藏需求。
不能绑定 reference 的私有实现格式。
不能让测试依赖某个模型的偶然失败。
```

## 6. 哪些环节可以自动化，哪些需要人类介入

可以交给大模型或 agent 的部分：

```text
1. 阅读 README 和源码结构，提取候选功能模块。
2. 草拟 PRD 初版。
3. 从 PRD 生成 requirement_map 初版。
4. 生成 unit/system case 草案。
5. 编写 reference 初版。
6. 跑模型 candidate 并汇总 score report。
7. 根据失败 case 生成改进建议。
```

需要人类重点介入的部分：

```text
1. 判断项目是否真实、有价值、不过于玩具化。
2. 决定抽象边界，防止任务过大或过小。
3. 检查 PRD 是否自然，是否像真实产品需求。
4. 检查 rubric 是否只是 PRD 翻译。
5. 判断失败 case 是否公平，是否能在 PRD 中找到原型。
6. 判断 gap 是否真实反映系统能力，而不是测试陷阱。
```

可以用 LLM-as-judge 辅助，但不能完全替代人工：

```text
LLM-as-judge 可以检查：
- rubric case 是否能回溯到 PRD。
- PRD 是否泄露过细测试细节。
- case 是否属于 unit 或 system。
- 失败分析是否合理。

人工最终确认：
- 抽象是否忠于真实 repo。
- 测试是否公平。
- 是否适合作为 benchmark 收录。
```

## 7. Multi-Agent 构造管线

可以设计成 8 个 agent：

```text
RepoAnalyzer Agent
职责：阅读 GitHub 仓库，提取核心功能、状态点、错误点和可抽象任务范围。

AbstractionPlanner Agent
职责：决定保留什么、简化什么、删除什么，形成 mini task 范围。

PRDWriter Agent
职责：生成自然的 prd.md，避免写成测试翻译。

RequirementMapper Agent
职责：把 PRD 需求编号化，生成 requirement_map.md。

RubricDesigner Agent
职责：生成 unit/system case 草案，并标注 system_dimension。

ReferenceBuilder Agent
职责：根据 PRD 和 rubric 写 reference，并保证 reference 100%。

Evaluator Agent
职责：运行 reference 和多个模型 candidate，生成 score report。

GapImprover Agent
职责：分析失败点，提出 system case 增强方案。
```

其中最关键的是：

```text
PRDWriter Agent:
防止 PRD 写成测试清单。

RubricDesigner Agent:
防止 rubric 脱离 PRD 或只做逐条翻译。

GapImprover Agent:
根据真实模型失败增强系统组合测试。

Human Reviewer:
最终判断公平性、真实性和是否可提交。
```

## 8. Agent system prompt 草案

### RepoAnalyzer Agent

```text
You analyze a real GitHub repository and identify benchmarkable behavior.
Focus on public behavior, state transitions, error recovery, and cross-feature interactions.
Do not propose a task unless it can be evaluated automatically.
Output: source signals, candidate mini task scope, core state model, and risks.
```

### PRDWriter Agent

```text
You write a public PRD for a benchmark task derived from a real repository.
The PRD must be natural and product-like.
Do not expose hidden tests, exact rubric cases, or overly specific edge-case sequences.
Describe capabilities, public behavior, invariants, error handling, non-goals, and evaluation style.
```

### RubricDesigner Agent

```text
You design rubric.json cases from a PRD and requirement map.
Every case must trace back to PRD requirements.
Do not merely translate each PRD sentence into one test.
Create unit cases for local behavior and system cases for composed behavior, state accumulation, error atomicity, operation order, and boundary crossing.
Avoid private-format traps and implementation-specific checks.
```

### Evaluator Agent

```text
You run reference and candidate implementations against the rubric.
Report unit score, system score, failed IDs, and likely failure causes.
Do not modify tests during evaluation.
Do not fabricate scores.
```

### GapImprover Agent

```text
You analyze candidate failures and propose rubric improvements.
Only propose cases that are clearly supported by the PRD.
Prefer system-level cases that combine existing public behaviors.
Reject hidden requirements, exact-text traps, environment-specific assumptions, and reference-private checks.
```

## 9. 判断一个任务是否合格

一个比较完整的 benchmark task 至少要满足：

```text
1. 来源是真实 GitHub 项目。
2. PRD 自然，不是 rubric 的翻译。
3. rubric 全部能回溯到 PRD 原型。
4. unit 和 system 分层清楚。
5. reference 能通过全部测试。
6. 至少一个模型出现 unit > system 的明显 gap。
7. score report 是真实跑出来的，不伪造。
8. 文件结构符合 Bmk-dev 官方格式。
```

最终可以记住这句话：

```text
PRD 写公开能力，rubric 写验收场景。
好的 system case 不是隐藏需求，而是把 PRD 中的多个自然行为组合起来验证。
```
